Poking at MAVLink

What happens when a drone has no idea who it's talking to

I learnt building drones back in college, then I learnt how to fly them, the natural next step was to learn to understand how they communicate, not just at an abstract high-level, but at the protocol level. This write up is a result of this rabbit hole and a documentation of the things I learnt.

Simulator ArduPilot SITL v4.8.0-dev
Protocol MAVLink v2 (unprotected)
Environment Ubuntu 22.04 VM, loopback only
Tools MAVProxy, pymavlink, tshark
01 — The Setup

Getting a drone online without buying one

The first challenge was the how. I wasn't entirely fond of the idea of running tests on a physical drone, primarily because I did not want to get into trouble and also because I did not have drone parts to spare. So instead, I relied on good old simulations (what would we ever do without them).
Anyway, my solution here was to use ArduPilot SITL (Software in the Loop), this essentially is the Ardupilot flight controller firmware, compiled to run as a native Linux process. The simulated drone here behaves identical to real hardware (except that you it won't fly into your head and give you a concussion).
The sensors synthesize telemetry data, it processes commands normally, responds to MAVLink packets just like a physical unit.

The stack used: ArduCopter SITL listening on TCP port 5760, with MAVProxy serving as both the ground station and protocol multiplexer. MAVProxy's multiplexing function is worth noting: it rebroadcasts the single SITL TCP stream to multiple UDP endpoints (14550 and 14551), allowing several clients to connect simultaneously. This mirrors a real field deployment, where one radio link comes in and feeds multiple ground tools. It also sets up the conditions for the attack scenarios in later sections.

ArduPilot SITL v4.8.0-dev MAVProxy 1.8.74 pymavlink 2.4.49 tshark / Wireshark Ubuntu 22.04

When MAVProxy connects, the drone initialises: sensors calibrate, parameters load, GPS lock is acquired. What is immediately notable is how much information the drone broadcasts before any commands are issued:

02 — The Protocol

What MAVLink is, and what it isn't

MAVLink (Micro Air Vehicle Link) is the telemetry and command protocol used by most open-source drone platforms, including ArduPilot and PX4, as well as a significant portion of the commercial market outside of DJI. It is a compact binary protocol designed to run over constrained radio links, originally optimized for bandwidth efficiency on low-power hardware. It achieves that goal well. Authentication was a secondary consideration, which shows.

MAVLink v1 has no authentication, no encryption, and no message signing. MAVLink v2 introduced optional message signing. The word optional carries most of the security weight here.

The first thing I examined was what the raw traffic looks like on the wire. tshark understands UDP natively but has no MAVLink protocol dissector, so the content inside UDP payloads appears as a raw hex stream. That is itself informative:

EVERY PACKET STARTS WITH 0xFD (MAVLink v2 MAGIC BYTE). THE TWO BYTES AFTER ARE FLAGS — BOTH ZERO MEANS NO SIGNING. THE SEQUENCE BYTE INCREMENTS 0x10, 0x11, 0x12 — ENTIRELY PREDICTABLE. THE MESSAGE ID TELLS YOU WHAT TYPE OF MESSAGE IT IS.

Several things are immediately apparent. Each packet begins with 0xFD, the MAVLink v2 start byte, also referred to as the magic byte or STX (start-of-text) byte. This byte serves as a frame delimiter and identifies the protocol version in use. MAVLink v1 uses 0xFE for the same purpose. The difference matters: v1 and v2 have different packet structures and different security capabilities, so a receiver uses this byte to determine how to parse what follows. The presence of 0xFD here confirms the session is running MAVLink v2. The two bytes immediately after are the incompatibility flags. Both are zero, meaning signing is disabled. In a signed packet, bit 0 of the first flag byte would be set to 1. The sequence byte increments from 0x10 to 0x11 to 0x12 and rolls over at 255. Any observer watching this stream can predict the next sequence number exactly.

The Injection Window

Sequence numbers are not a security feature. They exist for packet loss detection. Because they are predictable, however, they create an injection window: if a crafted packet reaches the drone before the next legitimate packet from the ground station, the drone processes it first. There is no cryptography to defeat. Just timing.

03 — Passive Interception

What the drone broadcasts without being asked

The first test was purely passive: connect a listener to the MAVProxy UDP port and record whatever the drone was transmitting. No commands were sent and no interaction was initiated. The following is what was broadcast continuously, without any prompting:

PASSIVE LISTENER — READING THE DRONE'S BROADCAST STREAM
Waiting for heartbeat... Connected! System: 1, Component: 0 ATTITUDE {time_boot_ms: 58114, roll: 0.0011552, pitch: 0.0009716, yaw: -0.0911482, rollspeed: 0.00036, pitchspeed: 0.00027, yawspeed: 0.00223} GLOBAL_POSITION_INT {time_boot_ms: 58114, lat: -353632610, lon: 1491652300, alt: 584070, relative_alt: -11, vx: -1, vy: 1, vz: 0, hdg: 35478} VFR_HUD {airspeed: 0.0217, groundspeed: 0.0217, heading: 354, throttle: 0, alt: 584.07, climb: -3.1e-05} SYS_STATUS { voltage_battery: 12600, current_battery: 0, battery_remaining: 100, drop_rate_comm: 0, errors_comm: 0} AHRS2 {roll: 0.0019, pitch: 0.0017, yaw: -0.1277, altitude: 584.08, lat: -353632610, lng: 1491652299}
GPS COORDINATES (LAT/LON × 1E7), ALTITUDE, HEADING, BATTERY LEVEL, ATTITUDE — ALL BROADCAST CONTINUOUSLY. NO REQUEST NEEDED. NO CREDENTIALS NEEDED.

The drone broadcasts this telemetry on every heartbeat cycle, approximately once per second. GPS position, altitude, heading, battery percentage, flight mode, and velocity are all transmitted continuously to any client connected to the port. On a real drone operating over radio telemetry at 915 MHz, a software-defined radio (SDR) receiver within range captures all of this without being on the same network. The radio link is inherently broadcast.

The coordinates shown are the SITL default home position in Canberra, Australia. lat: -353632610 corresponds to -35.3632610°. I learnt that MAVLink represents geographic coordinates as scaled integers (multiply by 1e7) to avoid floating-point inconsistencies across heterogeneous hardware. That yields positioning resolution on the order of 10mm, broadcast continuously to any observer (which ofcourse is not this accurate all the time due to sensor faults, but oh well..).

04 — Command Injection

Arming the drone from a rogue connection

This was the central question. With MAVProxy connected as the legitimate ground station and the drone running, I opened a second connection to the same UDP port using a separate client and sent commands directly. There is no handshake, no pairing process, and no authorization check. The second client is treated identically to the first.

ROGUE CLIENT — UNAUTHENTICATED ARM & TAKEOFF
connected to system 1 Arming Armed takeoff command sent
FOUR LINES. CONNECTED, SENT ARM COMMAND, GOT CONFIRMATION, SENT TAKEOFF. THE LEGITIMATE GCS HAD NO MECHANISM TO PREVENT THIS.

The legitimate ground station confirmed what happened on its end:

LEGITIMATE GROUND STATION — WHAT IT SAW
Got COMMAND_ACK: DO_SET_MODE: ACCEPTED GUIDED> Mode GUIDED Got COMMAND_ACK: COMPONENT_ARM_DISARM: ACCEPTED AP: Arming motors ARMED Arming checks disabled Got COMMAND_ACK: NAV_TAKEOFF: ACCEPTED AP: EKF3 IMU0 MAG0 in-flight yaw alignment complete AP: EKF3 IMU1 MAG0 in-flight yaw alignment complete Flight battery 100 percent height 15
THE GCS SAW THE MODE SWITCH, THE ARM, AND THE TAKEOFF AS LEGITIMATE EVENTS. IT HAD NO WAY TO KNOW THEY CAME FROM A SECOND CLIENT. THE DRONE IS NOW AT 15 METRES.

The sequence was: force the drone into GUIDED mode (the flight mode that accepts external position commands, required for autonomous scripted navigation), then arm the motors, then issue a takeoff command. COMMAND_ACK is the drone's acknowledgment message, confirming it received and accepted a command. The drone sent COMMAND_ACK: ACCEPTED for all three steps. The legitimate ground station had no mechanism to distinguish these commands from its own.

Why This Works

MAVLink identifies the sender of a message by system ID and component ID. These values are set by the client itself; the drone does not assign or verify them. Any client can declare any system ID and will be treated accordingly. The flight controller processes commands in the order it receives them, validates only the message format, and replies with an ACK. There is no concept of a trusted source list.

05 — Full Attack Chain

Mode force, disarm, position redirect

The arm and takeoff scenario is demonstrative, but the more operationally significant case is mid-mission disruption. A second test: connect to a drone already in flight, force a mode change, inject a disarm command, and redirect its position target.

FULL CHAIN — INTERCEPT, DISARM, REDIRECT
target acquired - drone is system 1 guided mode forced - intercepted disarm injected
IDENTIFY THE TARGET'S SYSTEM ID FROM THE HEARTBEAT, FORCE A MODE SWITCH, DISARM. THREE STEPS, NO CREDENTIALS.
POSITION REDIRECT — MAKE IT GO SOMEWHERE ELSE
current position: -353632609, 1491652298, altitude 584.1m Current position: -35.3632609, 149.1652298, altitude 0.03m moving north
READ THE DRONE'S CURRENT COORDINATES FROM ITS BROADCAST. SEND A SET_POSITION_TARGET_GLOBAL_INT MESSAGE WITH NEW COORDINATES. THE DRONE MOVES. THE REAL GCS WATCHES IT DEVIATE FROM MISSION WITH NO EXPLANATION.

The SET_POSITION_TARGET_GLOBAL_INT message includes a type_mask bitmask that specifies which fields the drone should act on from position, velocity, and acceleration. Setting the appropriate bits and supplying modified coordinates causes the drone to treat the message as a legitimate navigation command. The flight controller does not verify that the new target is consistent with an active mission, nor does it check whether the source sending the command is the same system that originated the mission.

06 — The Defense Side

MAVLink v2 signing — what it actually does

MAVLink v2 includes an optional message signing mechanism. The implementation appends a 48-bit truncated HMAC-SHA256 signature to each packet, computed using a shared secret key pre-configured on both the drone and the ground station. Any packet sent without the correct key fails verification on the receiving end. With signing enabled on the simulated drone:

SIGNING DEMO — LEGITIMATE VS ROGUE PACKET
secret key is 0e5b391c74227cf4c19cf6a5c2e25503a57ec3686df9f44572e642e409350b72 Connected to system 1 Signature: c3f7503ffd18 Legitimate packet verified: True Attacker packet verified: False
WITH SIGNING ON, A PACKET WITHOUT THE CORRECT KEY IS REJECTED. THIS IS THE CORRECT BEHAVIOUR. THE PROBLEM IS THAT MOST DEPLOYMENTS NEVER TURN IT ON.

The mechanism works as designed. A packet without the shared key is rejected. The practical deployment problems, however, are substantial:

  • 01 It is optional and off by default. For backward compatibility, a MAVLink v2 drone will accept unsigned connections if the connecting client requests it. An attacker can explicitly ask to skip signing. The drone complies.
  • 02 The signature is 48 bits, not 256. It is a truncated HMAC-SHA256 signature. 48 bits is workable for message integrity, as the collision probability is low enough for practical use, but it is a constraint worth noting.
  • 03 Key distribution is a solved problem that nobody has solved for drones. The shared secret must be distributed to both the ground station and the drone before any signed session can begin. There is no standardized provisioning flow for this. Most operators do not configure signing because the process is non-trivial and underdocumented.
  • 04 Signing doesn't encrypt telemetry. Even with signing on, the position, battery, and attitude broadcasts are readable by anyone. Signing prevents command injection; it does nothing for telemetry privacy.

A more operationally resilient defense is anomaly detection at the application layer: a ground station configured to flag mode switches it did not initiate, arm commands from unexpected system IDs, or GPS coordinates inconsistent with inertial measurements. This approach requires no key exchange and no protocol changes. It requires only that the operator be alerted when something anomalous occurs. Geofencing operates on a similar principle: if the flight controller is configured to reject coordinates outside a defined boundary, a redirected position command fails regardless of whether the attacker holds the signing key.

07 — What I Took Away

The interesting parts

The attack scenarios documented here were not particularly surprising to anyone familiar with the MAVLink specification. The authentication model is known to be weak. The more interesting question is why these conditions persist in production deployments.

The fix exists. MAVLink v2 signing is implemented in the firmware and supported by every major ground station. The mechanism works, as the signing demo confirmed. The problem is that enabling it is opt-in, requires a non-standardized pre-shared key provisioning step, and breaks backward compatibility with older tooling. Given that friction, most operators leave it off. That is not an engineering failure in the traditional sense. The engineers built the capability and made it available. It is a product and deployment failure: no one made enabling signing the default or the path of least resistance.

What I find most interesting here, from a product perspective, is that the security gap is not in the specification. It is in defaults, onboarding, and the friction of doing the right thing. A drone company could fix most of this unilaterally, without any changes to the MAVLink protocol, just by shipping firmware that refuses unsigned connections unless explicitly overridden. They haven't. That's a product decision, not a technical limitation.
